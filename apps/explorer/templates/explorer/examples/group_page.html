This page has two levels: a sidebar listing all groups and a main area showing models for the selected group. Clicking a group in the sidebar navigates to that group's page. Here's how to build the same pattern in your own app.

1. URLs

One route handles every group. The <str:group> capture becomes self.kwargs["group"] in the view, which the mixin passes to the registry to look up models.

apps/myapp/urls.py 
from django.urls import path
from .views import GroupDashboardView

# Pattern:
#   /dashboard/<group>/
#
# <str:group> captures the group name from the URL.
# The mixin reads it via self.kwargs["group"] and passes
# it to explorer_registry.get_group_context(group_name),
# which returns all models assigned to that group.
#
# The group name comes from explorer_group on ModelAdmin:
#   explorer_group = "Monitoring"  →  /dashboard/Monitoring/
#   explorer_group = "Admin"       →  /dashboard/Admin/
#
# Example: visiting /dashboard/Monitoring/ returns:
#   - group_name:  "Monitoring"
#   - models:      [Heartbeat, HeartbeatEpoch, HeartbeatDaily, ...]
#   - all_groups:  ["Admin", "Monitoring", ...]

urlpatterns = [
    path(
        "dashboard/<str:group>/",
        GroupDashboardView.as_view(),
        name="myapp-group-dashboard",
    ),
]

2. View — with mixin (recommended)

ExplorerGroupMixin handles everything: reads the group from the URL, calls the registry, 404s if not found, and populates group_name, models, and all_groups into the template context. Your view is just two lines.

apps/myapp/views.py 
from django.views.generic import TemplateView
from apps.smallstack.mixins import StaffRequiredMixin
from apps.explorer.mixins import ExplorerGroupMixin


# Dynamic — reads group from URL parameter
class GroupDashboardView(ExplorerGroupMixin, StaffRequiredMixin, TemplateView):
    template_name = "myapp/group_dashboard.html"


# Hardcoded — always shows "Monitoring", no URL param needed
class MonitoringView(ExplorerGroupMixin, StaffRequiredMixin, TemplateView):
    template_name = "myapp/monitoring.html"
    explorer_group = "Monitoring"

2b. View — manual (if you need custom logic)

Use get_group_context() directly if you need to add extra data or transform the results. Call .as_context() to get a dict ready for context.update().

apps/myapp/views.py 
from django.http import Http404
from django.views.generic import TemplateView
from apps.smallstack.mixins import StaffRequiredMixin
from apps.explorer.registry import explorer_registry


class GroupDashboardView(StaffRequiredMixin, TemplateView):
    template_name = "myapp/group_dashboard.html"

    def get_context_data(self, **kwargs):
        context = super().get_context_data(**kwargs)
        ctx = explorer_registry.get_group_context(self.kwargs["group"])
        if not ctx:
            raise Http404("Group not found.")
        context.update(ctx.as_context())
        # Add your own extra data here
        context["my_custom_thing"] = "whatever you need"
        return context

3. Template

The sidebar loops all_groups to build navigation links. The main area loops models to render cards. Each model is a ModelCardInfo dataclass with attributes like .verbose_name_plural, .count, .list_url, .app_label, and .readonly.

4. Register models with a group

Set explorer_enabled = True on any ModelAdmin. Use explorer_group to assign a group name — this is the value that appears in the sidebar and matches the URL parameter. Without it, the group defaults to the app label.

apps/heartbeat/admin.py 
from django.contrib import admin
from .models import Heartbeat, HeartbeatEpoch, HeartbeatDaily


@admin.register(Heartbeat)
class HeartbeatAdmin(admin.ModelAdmin):
    list_display = ("checked_at", "status", "response_time_ms")

    # Explorer integration
    explorer_enabled = True          # register with Explorer
    explorer_group = "Monitoring"    # sidebar group name

    # Optional overrides:
    # explorer_fields = [...]        # override list columns
    # explorer_readonly = True       # force read-only mode
