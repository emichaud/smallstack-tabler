This is the most common use case — a page dedicated to one model with a full CRUD experience: list, detail, create, update, delete. The mixin provides everything the crud_table template tag needs, so you get a working data table with zero queryset or field configuration in your view.

1. URL

Two URL parameters identify the model: app_label and model_name. These match the model's _meta.app_label and _meta.model_name in Django.

apps/myapp/urls.py 
from django.urls import path
from .views import ModelPageView

# Pattern:
#   /models/<app_label>/<model_name>/
#
# <app_label> and <model_name> identify the model in the
# Explorer registry. These are the same values Django uses
# internally: Model._meta.app_label, Model._meta.model_name.
#
# The mixin reads them via self.kwargs and passes them to
# explorer_registry.get_model_context(app_label, model_name),
# which returns everything needed to render the CRUD table.
#
# Example: /models/heartbeat/heartbeat/ returns:
#   - object_list:       queryset of all Heartbeat rows
#   - list_fields:       ["checked_at", "status", "response_time_ms"]
#   - url_base:          "explorer/heartbeat/heartbeat"
#   - crud_actions:      [LIST, DETAIL, CREATE, UPDATE, DELETE]
#   - create_view_url:   "/smallstack/explorer/heartbeat/heartbeat/create/"
#   - link_field, field_formatters, detail_fields, etc.

urlpatterns = [
    path(
        "models/<str:app_label>/<str:model_name>/",
        ModelPageView.as_view(),
        name="myapp-model-page",
    ),
]

2. View — with mixin (recommended)

ExplorerModelMixin reads app_label and model_name from the URL, calls the registry, and populates the template context with everything crud_table needs. Your view is just two lines.

apps/myapp/views.py 
from django.views.generic import TemplateView
from apps.smallstack.mixins import StaffRequiredMixin
from apps.explorer.mixins import ExplorerModelMixin


# Dynamic — reads app_label and model_name from URL
class ModelPageView(ExplorerModelMixin, StaffRequiredMixin, TemplateView):
    template_name = "myapp/model_page.html"


# Hardcoded — always shows Heartbeat, no URL params needed
class HeartbeatPageView(ExplorerModelMixin, StaffRequiredMixin, TemplateView):
    template_name = "myapp/heartbeats.html"
    explorer_app_label = "heartbeat"
    explorer_model_name = "heartbeat"

2b. View — manual (if you need custom logic)

Use get_model_context() directly when you need to filter the queryset, add extra context, or combine multiple models on one page.

apps/myapp/views.py 
from django.http import Http404
from django.views.generic import TemplateView
from apps.smallstack.mixins import StaffRequiredMixin
from apps.explorer.registry import explorer_registry


class HeartbeatPageView(StaffRequiredMixin, TemplateView):
    template_name = "myapp/heartbeats.html"

    def get_context_data(self, **kwargs):
        context = super().get_context_data(**kwargs)
        ctx = explorer_registry.get_model_context("heartbeat", "heartbeat")
        if not ctx:
            raise Http404("Model not found.")
        context.update(ctx.as_context())
        # Add your own extra data here
        context["my_custom_thing"] = "whatever you need"
        return context

4. What you get

With the mixin and crud_table, you get a complete CRUD experience out of the box. No need to define querysets, forms, or URL routing for individual actions — Explorer has already generated all of that from your admin registration.

What the mixin provides in context 
# Context variables populated by ExplorerModelMixin:
#
# object_list              → queryset of all model rows
# list_fields              → columns for the table (from admin list_display)
# detail_fields            → fields shown on the detail page
# link_field               → which column links to the detail view
# url_base                 → e.g. "explorer/heartbeat/heartbeat"
# crud_actions             → [LIST, DETAIL, CREATE, UPDATE, DELETE]
# field_formatters         → custom display formatters (if any)
# create_view_url          → URL for the "Add" button (None if readonly)
# object_verbose_name      → "Heartbeat"
# object_verbose_name_plural → "Heartbeats"
# model_info               → ModelCardInfo dataclass with count, list_url, etc.
#
# The crud_table tag reads these automatically.
# Detail, create, update, and delete views are already
# registered by Explorer — links just work.

5. Register models in admin

This is the same admin setup used for group pages. Set explorer_enabled = True and Explorer handles the rest. The list columns come from list_display unless you override with explorer_fields.

apps/heartbeat/admin.py 
from django.contrib import admin
from .models import Heartbeat

@admin.register(Heartbeat)
class HeartbeatAdmin(admin.ModelAdmin):
    list_display = ("checked_at", "status", "response_time_ms")

    # Explorer integration
    explorer_enabled = True          # register with Explorer
    explorer_group = "Monitoring"    # group for the group page

    # Optional overrides:
    # explorer_fields = [...]        # override list columns
    # explorer_readonly = True       # force read-only (hides create/edit/delete)
